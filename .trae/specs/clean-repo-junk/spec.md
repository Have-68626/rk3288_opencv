# 仓库垃圾文件清理（可回滚）Spec

## Why
在重构前先清理仓库中的垃圾与过期产物，降低仓库体积与噪声，避免将无价值文件带入后续重构与版本历史。

## What Changes
- 新增“一键扫描与清理”脚本（Node.js 优先，必要时可兼容 PowerShell 调用方式），用于识别并处理常见垃圾文件与构建缓存。
- 新增可配置规则（白名单/黑名单、保护规则、确认策略、输出目录等），支持 dry-run 模式。
- 新增清理前后快照生成：文件树快照与磁盘占用报告（可对比）。
- 新增回滚机制：删除前先隔离（移动到临时隔离区），可一键恢复；隔离区可配置且默认位于系统临时目录，避免污染仓库。
- 新增清理结果输出：清理清单（逐项文件/目录、大小、原因、是否隔离/删除）与释放空间统计。
- 新增 CI 校验：在 CI 中验证清理后依赖安装、构建、测试与运行流程仍可通过；并在 CI 中执行脚本的 dry-run/规则一致性校验。
- 视需要补充忽略规则（如 .gitignore）以防垃圾文件再次进入版本库。
- **BREAKING（可选）**：若仅清理无法达到“仓库体积减少≥30%”，可能需要进行 Git 历史瘦身（重写历史）以真正降低远端仓库体积；该步骤必须单独开关、强制备份、并在执行前获得明确确认。

## Impact
- Affected specs: 仓库卫生、构建产物管理、CI 验证链路、可回滚清理能力
- Affected code: 新增 scripts/ 或 tools/ 下清理脚本与配置；可能修改 .gitignore；可能修改 CI 配置（如 GitHub Actions / 其他流水线）

## ADDED Requirements

### Requirement: 扫描与清理脚本
系统 SHALL 提供一个可在项目根目录运行的脚本，用于扫描并处理垃圾文件与过期产物，满足以下能力：
- 支持 dry-run（默认开启）：仅输出计划处理项，不做任何修改。
- 支持真实执行：在用户确认后才进行文件操作。
- 支持可配置白名单：白名单优先级最高，命中的路径/模式永不删除/隔离。
- 支持可配置规则集：包含“目标匹配规则”（要清理的内容）与“保护规则”（绝不触碰的内容）。
- 支持交互确认与非交互模式（如 --yes / CI 模式）。
- 支持回滚：真实执行时先隔离（移动）到隔离区，并生成可用于恢复的清单；提供 restore 子命令或等价能力恢复到原位置。

#### Scenario: dry-run 成功
- **WHEN** 用户运行脚本并指定 dry-run（或默认）
- **THEN** 输出候选清理项列表、预估释放空间、以及“不会进行任何删除/移动”的提示
- **AND** 生成“清理前快照”与“计划清理清单”

#### Scenario: 真实清理成功（可回滚）
- **WHEN** 用户以非 dry-run 方式运行并确认执行
- **THEN** 所有命中项先移动到隔离区（或按配置直接删除，但默认隔离）
- **AND** 生成“清理后快照”“实际清理清单”“释放空间统计”
- **AND** 提供 restore 能力可恢复到清理前状态

#### Scenario: 保护规则生效
- **WHEN** 命中项疑似为业务源码或关键工程文件
- **THEN** 脚本必须拒绝处理并输出原因，除非该路径被明确加入白名单/允许列表（显式 opt-in）

### Requirement: 需要覆盖的垃圾类型
系统 SHALL 能识别并处理下列类别（按规则集逐步扩展，先覆盖高置信度类别）：
- 过期构建产物：dist/、build/、out/、coverage/、.next/、.turbo/ 等（以仓库实际技术栈为准）
- IDE/工具缓存：.idea/（缓存类）、.vscode/（可选）、.gradle/、.cxx/、*.iml 等（仅在确认其为可重建且不应入库时）
- Node 生态缓存：node_modules/.cache、.eslintcache、*.tsbuildinfo 等（若仓库存在 Node 工具链）
- 日志与临时文件：*.log、*.tmp、*.temp、*.bak、*.old、*.swp 等（排除明确需要保留的日志目录）
- 系统隐藏垃圾：.DS_Store、Thumbs.db、desktop.ini 等
- 未引用资源文件（可选增强）：图片/字体/样式等静态资源的“未引用扫描”只作为建议清单输出，默认不自动删除，需显式开启并结合白名单

### Requirement: 快照与报告
系统 SHALL 在清理前后生成以下产物（路径可配置，默认输出到 ErrorLog/cleanup/ 下；若不存在则创建）：
- 文件树快照：清理前/清理后各一份（可读文本或 JSON）
- 磁盘占用报告：清理前/清理后各一份（总量与前 N 大目录/文件）
- 对比摘要：释放空间、处理项数量、失败项与原因

### Requirement: 可测量的“仓库体积”指标
系统 SHALL 明确并输出至少两类体积指标，用于判定“减少≥30%”：
- 工作区体积（不含 .git）：用于反映开发者实际占用
- 版本库体积（可选）：例如 .git 目录体积或打包后的对象体积（用于反映历史是否真的变小）

### Requirement: CI 验证（清理后仍可用）
系统 SHALL 在 CI 中完成以下验证：
- 运行清理脚本（dry-run/规则校验）确保脚本可用且规则一致
- 执行项目既有的依赖安装、构建、测试与运行（以仓库现有流程为准）
- 若 CI 无法覆盖“运行”，至少覆盖“构建 + 测试”

## MODIFIED Requirements
无（本变更为新增能力与仓库治理流程）

## REMOVED Requirements
无
