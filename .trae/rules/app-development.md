#适用范围

- 本规则仅对“程序开发项目/代码仓库”生效（如 GitHub 下的开发仓库）。
#交流与文档规范

- 语言限制：全过程（含代码注释、文档、回复）必须使用 ZH_CN。
- 必备文件：项目根目录须包含 README.md、DEVELOP.md（开发文档）及 CREDITS.md（致谢与许可证）。
  - 若缺失：先提醒我；在我明确同意后再创建/补齐。
- 配置记录：在 DEVELOP.md 中实时维护目录树，并记录版本号、应用名、环境变量等频繁变动项的修改路径。
- 资产审计：第三方内容与依赖须在 CREDITS.md 注明来源及许可证。
#代码与依赖标准

- 质量要求：坚持高可读、易维护、低冗余；关键逻辑必须附带合理注释。
- 依赖申明：明确项目环境要求。若为模糊需求或实验性开发，须显式申明新增依赖及其推荐理由。
- 便携性原则：严禁硬编码路径。必须使用标准包管理工具及配置文件（如 .env），并提供环境一键复现指南。
- 单语言原则：严禁在同一文件中混合使用多种编程语言（如非必要，避免 HTML/CSS/JS 混写或 Python 嵌入大量 C 字符串）。
- 高度模块化：严格遵循单一职责原则，不同功能逻辑必须拆分到独立文件。
#目录与环境管理

- GitHub 映射：在 DEVELOP.md 中标注哪些内容为程序本体、哪些需上传 GitHub。
- 错误驱动：强制建立 ErrorLog 文件夹，作为用户报错内容的指定存放点，用于调试分析。
- 临时文件清理：静态链接源文件统一下载至 D:\ProgramData\Temp。任务结束后必须提醒我删除临时源文件并确认清理。
#其他

- 注意检查是否有合适的 MCP 工具、Skills，联系实际后再决定是否利用它们。
- 开发相关操作尽量优先调用已建好的开发智能体（Git/GitHub、文件、自动化测试等），避免默认助手乱用工具。
