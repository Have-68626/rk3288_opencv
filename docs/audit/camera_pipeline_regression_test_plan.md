# 相机链路兼容性与回归测试计划（audit-camera-pipeline）

## 0. 目标与验收口径
- 目标：验证“权限 → 初始化 → 预览帧拉取 → 生命周期释放/恢复 → 错误可恢复”不回归，并覆盖关键兼容性差异。
- 核心验收：
  - 不崩溃：启动/停止/切后台/旋转/USB 插拔等场景无 ANR/Crash
  - 不占用：退后台后不再占用摄像头（回前台可恢复）
  - 不阻塞：预览帧拉取不阻塞主线程（交互不明显卡顿）
  - 不泄漏：长时间运行内存不持续上涨（允许小幅波动）

## 1. 覆盖范围（系统/设备）

### 1.1 Android 版本覆盖
- Android 5.0/5.1（minSdk 21 兼容性）
- Android 7.x（RK3288 常见系统）
- Android 10（后台限制增强后的稳定性）
- Android 13/14（权限/后台限制/前台服务策略基线验证）

### 1.2 设备矩阵（建议最小集）
- RK3288 板卡（至少 1 台）：主摄/USB 摄像头各 1 次验证
- 主流手机 2 台：一台高端（多摄），一台中低端（内存更紧）
- 可选：Android 模拟器（仅验证 UI/权限逻辑，不依赖真实相机时可跑）

## 2. 手工回归用例清单

### 2.1 权限流（合规）
- 首次启动：
  - 未授权 → 弹窗申请相机权限 → 允许 → 可启动监控
  - 未授权 → 弹窗申请相机权限 → 拒绝 → 进入安全模式（按钮禁用/提示）
- 永久拒绝（不再询问）：
  - 触发申请后 → 弹设置引导 → 跳转设置授权 → 返回后状态正确刷新

### 2.2 生命周期释放/恢复
- 运行监控中按 Home：
  - 预期：自动停止监控；回前台自动恢复（如权限仍满足）
- 运行监控中切到多任务界面：
  - 预期：同上
- 锁屏/解锁：
  - 预期：解锁后可恢复，不崩溃

### 2.3 启停与并发健壮性
- 连续快速点击 START/STOP（至少 30 次）：
  - 预期：不崩溃；不出现“卡死无法再启动”
- 切换摄像头下拉框（真实相机之间切换）：
  - 预期：自动 stop → re-init → 可重新 start
- USB 摄像头插拔（若设备支持）：
  - 预期：相机列表刷新；在运行中插拔不崩溃（必要时进入可恢复状态）

### 2.4 预览帧链路（性能/背压）
- 运行监控 5 分钟：
  - 预期：界面滑动/按钮响应正常（无明显卡顿/ANR）
- 长时间运行 30 分钟：
  - 预期：内存无持续线性增长；FPS/CPU/MEM 指标可持续刷新

### 2.5 Mock 源链路
- 文件选择（图片/视频）：
  - 预期：加载成功后可启动监控；停止后可再次选择
- 系统相机拍照 Mock：
  - 预期：拍照返回后可作为 Mock 源使用；多次重复无异常

## 3. 自动化建议（能做则做）

### 3.1 单元测试（JVM）
- `:app:testDebugUnitTest`：
  - 保持现有测试通过（设备分类/资源审计）
  - 建议新增：权限状态机的纯逻辑测试（不依赖 Android Framework 时）

### 3.2 Instrumentation（建议路线）
由于相机依赖硬件且回放难，推荐两条路径择一推进：
- 路线 A（优先）：将“帧来源”抽象为接口（FrameSource），instrumentation 使用假帧源回放；只测 UI/生命周期/恢复口径。
- 路线 B：引入 `androidx.camera:camera-testing`（若未来切到 CameraX），使用 FakeCamera 验证 bind/unbind 与异常恢复。

## 4. 观测与取证
- 日志导出：App 内 Log Viewer 导出 ZIP（含 manifest.txt CRC32）用于回归留档。
- 关键取证点：
  - 权限状态变化、start/stop、onStop/onStart 自动恢复
  - 首帧到达时间（可结合日志与肉眼观察）
