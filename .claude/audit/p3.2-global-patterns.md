## 2. 命名一致性

| 方面 | 评价 |
|:-----|:------|
| 类命名 (PascalCase) | ✅ 一致 |
| 文件名与类名 | ⚠️ 良好，Config.h 为 namespace 非类 |
| **命名空间** | 🔴 **5 种策略混用** — 全局/rk_win/rk_wcfr/rk_accel/rklog，cpp 核心引擎完全无 namespace |
| 枚举 (enum class) | ✅ 一致，12+处全部使用 enum class |
| 常量命名 | ⚠️ 混用 — ALL_CAPS(Config.h) vs kCamelCase(HttpFacesServer) vs kPascalCase(Engine.h) |
| 成员变量 | ⚠️ 混用 — trailing_underscore(win模块) vs camelCase(cpp模块) |

## 3. 错误处理模式

| 模式 | 使用区域 | 评价 |
|:-----|:---------|:-----|
| 日志 + 返回 bool | Win模块 | 主导模式 |
| 结构化 JSON 错误码 | HttpServer | **最成熟** |
| `(void)` 丢弃返回值 | 40+处 | 🔴 约10处是真实错误吞咽（配置写入失败等）|
| RAII 守卫 | 仅 HttpFacesServer/AbcTestRunner | 其他 Windows API 调用处手动管理 CloseHandle |

## 4. 头文件包含

| 方面 | 评价 |
|:-----|:------|
| `#pragma once` | ✅ 所有项目头文件一致 |
| 包含顺序 | ⚠️ Win模块规范，cpp模块较随意 |
| forward declaration | ✅ Win模块大量使用，cpp模块几乎不用（Engine.h #include 6个头文件）|
| `extern "C"` | ✅ 仅第三方需要 |

## 5. 线程安全模式

## 6. Top 5 全局问题

| 排名 | 问题 | 影响 |
|:---:|:-----|:-----|
| **1** | **无公共工具库** — escapeJson(6)+jsonEscape(4)+utf8FromWide(4)+toLower/trim/nowEpochMillis/SocketGuard/WsaGuard = 8组22处重复 | 维护噩梦 |
| **2** | **cpp模块无统一 namespace** — 全局污染 vs rk_win/rk_wcfr/rk_accel/rklog 5种策略混用 | 命名冲突风险 |
| **3** | **`(void)` 返回值丢弃** — 40+处，约10处真实错误吞咽（配置写入/文件写入/加解密失败） | 无声数据丢失 |
